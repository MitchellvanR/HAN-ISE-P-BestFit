## 1. Inleiding

### 1.1 Doel van dit document
In dit document wordt het testplan beschreven voor het ISE-project. Het doel van dit testplan is om 
de aanpak en methodologie te beschrijven die zullen worden gebruikt om de kwaliteit van de software te garanderen en 
om ervoor te zorgen dat deze voldoet aan de gestelde eisen. Het testplan beschrijft de teststrategie, de testomgeving en
de testgevallen. 

### 1.2 Belanghebbenden
Dit testplan is bedoeld voor alle belanghebbenden van het project, waaronder het projectteam en de toekomstige 
ontwikkelteams en onderhouders. 
Het testplan is gebaseerd op de vereisten van het project en de doelstellingen van de tests.

## 2 Teststrategie

### 2.1 Stored procedures
Voor de stored procedures wordt de volgende strategie toegepast:

- Het scenario dat alles goed gaat, word getest.
- Voor elk scenario waarvan voorzien wordt dat de stored procedure fout kan gaan, wordt een aparte test geschreven die dit scenario test.

Zodra deze tests geschreven zijn, kunnen wij er zeker van zijn dat de stored procedure aan de functionaliteit voldoet
die wij en de opdrachtgever verwachten.

### 2.2 Triggers
Per trigger worden voor de volgende scenarios tests geschreven:

- Het scenario waar de trigger moet slagen.
- Elk scenario waarvan voorzien wordt dat de trigger moet falen.

Voor elk van deze tests wordt ook een versie geschreven waar er gebruik wordt gemaakt van een batch, zodat wij er zeker
van kunnen uitgaan dat de trigger niet alleen werkt voor één waarde.

### 2.3 Staging area
In eerste instantie waren we van plan om alle code van de staging area te testen. Voor een groot deel van het project
is dit zonder al te veel moeite gelukt. Helaas zijn we er later achter gekomen dat de tests voor de code die verantwoordelijk is
voor het genereren van testdata te complex zouden worden om te schrijven met onze kennis over python en pytest. Om deze
hebben we er voor gekozen om deze code niet te testen.

## 3 Testomgeving

### 3.1 Database

Om alle tests uit te voeren van de database, is een testomgeving nodig. De testomgeving die nodig is voor het uitvoeren van de tests
bestaat uit de aangeleverde Docker container waar de database en cache in draaien. Deze Docker container
start twee Microsoft SQL Server databases, een hoofddatabase en een testdatabase. Daarnaast start Docker een Redis 
cache. 

De Docker container is te vinden in de map `docker` in de root van het project. De Docker container kan worden
gestart met het commando `docker compose up` in deze map van het project. Zodra de Docker is opgestart, kunnen de 
testgevallen doorlopen worden.

Om de testgevallen uit te voeren, is een IDE of code editer nodig. De IDE die door de projectgroep gebruikt wordt voor 
het uitvoeren van de testgevallen is IntelliJ IDEA. In IntelliJ IDEA kan er connectie gemaakt worden met de database. 
De database is te vinden op `localhost:1433`. De gebruikersnaam is `LOG_BEST_FIT_ADMIN` en het wachtwoord is `BestFitAdminExamplePassword0?`. De database 
die gebruikt wordt voor de testgevallen is `Test_BestFit`. Zodra de connectie is gemaakt, kunnen de testgevallen 
doorlopen worden.

### 3.2 Staging area

Om alle test uit te voeren voor de staging area, is het nodig om alle dependencies te installeren. Dit kan gedaan worden
door het commando `poetry install` uit te voeren in de map `staging_area/bestfit` in de root van het project. Zodra alle
dependencies geïnstalleerd zijn, kunnen de testgevallen doorlopen worden.

## 4 Testgevallen voor stored procedures
Dit zijn alle testgevallen voor stored procedures. Elk testgeval is genummerd, zodat deze makkelijk terug te vinden zijn. 
Verder worden het testscenario, de input die wordt gegeven en het verwachte resultaat beschreven.
Er zijn drie stored procedures voor het genereren van history tabellen. 
Deze stored procedures zijn: sproc_generateHistoryTable, sproc_generateHistoryTableTrigger en sproc_generateHistoryTables.
Voor elk van deze procedures is er maar een test en deze zijn samen in de volgende tabel gezet:

| **Stored procedure**              | **Testscenario**                                        | **Input**                                | **Verwachtte resultaat**                                         |
|-----------------------------------|---------------------------------------------------------|------------------------------------------|------------------------------------------------------------------|
| sproc_generateHistoryTable        | Een geschiedenis tabel wordt gegenereerd.               | SPROC parameters: tableName = 'Employee' | De geschiedenistabel wordt zonder problemen aangemaakt.          |
| sproc_generateHistoryTableTrigger | Triggers bij een geschiedenis tabel worden gegenereerd. | SPROC parameters: tableName = 'Employee' | De insert en delete triggers worden zonder problemen aangemaakt. |
| sproc_generateHistoryTables       | Beide stored procedures worden goed aangeroepen.        | Geen data of parameters.                 | De stored procedures zijn zonder problemen aangeroepen.          |

Voor elke stored procedure en trigger is een tabel opgesteld met de testscenario's en de bijbehorende testdata en parameters van 
de stored procedure of insert data van triggers. Testdata die voor elk testscenario hetzelfde blijft, wordt opgenomen in een 
SetUp procedure die aan het begin van elke test uitgevoerd wordt. Om te zien welke testdata per stored procedure deze SetUp 
procedure instelt, is aan het begin van elke subkop in dit hoofdstuk een stuk tekst geschreven die de testdata toelicht.
